Critical data problems to watch for:
1. CLASS IMBALANCE — likely 2M safe vs 500K malicious. A naive model learns to
   say "safe" for everything and gets 80% accuracy while being useless. Fix: balance the

3. TEMPORAL LEAKAGE — testing on old data that existed during training. Fix: split by
   date — train on URLs before a cutoff date, test on URLs after.

4. DEAD URLs — malicious URLs from 2019 may no longer resolve. The URL string itself
   still carries signal — "paypal-verify-account.ru" is suspicious regardless.

--------------------------------------------------------
PHASE 2 — Data Preprocessing
--------------------------------------------------------

Step 2.1 — Cleaning
Remove these (they add noise):
- URLs longer than 2000 characters
- URLs with no TLD
- Localhost and private IPs (127.x.x.x, 192.168.x.x)
- Non-HTTP/HTTPS protocols
- Duplicates

  df = df[df['url'].str.startswith('http')]
  df = df[df['url'].str.len() < 2000]
  df = df.drop_duplicates(subset='url')

Step 2.2 — URL Normalization
Apply consistently to training data AND at inference time (they must match):
- Lowercase
- Remove trailing slash
- Decode percent-encoding (%20 → space)
- Strip utm_* tracking parameters

Step 2.3 — Handling Class Imbalance
  If ratio is < 3:1 (safe:malicious)  → no action needed
  If ratio is 3:1 to 10:1             → weighted loss function (easiest)
  If ratio is > 10:1                  → oversample minority OR undersample majority

  Weighted loss (recommended — no data loss):
    class_weights = [1.0, 3.0, 5.0]   # safe, suspicious, dangerous
    # Tell the model: getting a dangerous URL wrong costs more

Step 2.4 — Train / Validation / Test Split
  70% → training set     (model sees this during training)
  15% → validation set   (check progress, tune hyperparameters)
  15% → test set         (NEVER touch until final evaluation)

  GOLDEN RULE: The test set is sacred. Look at it once, at the very end, after
  all decisions are made. If you keep peeking and adjusting, you overfit to it
  and your published accuracy will be a lie.

  Use stratified split:
    from sklearn.model_selection import train_test_split
    train, temp = train_test_split(df, test_size=0.30, stratify=df['label'], random_state=42)
    val, test   = train_test_split(temp, test_size=0.50, stratify=temp['label'], random_state=42)

Step 2.5 — Feature Engineering
For BERT-class model (our approach): the tokenizer handles this automatically.
For character-level model: convert URL to character IDs, pad to fixed length (200 chars).

PHASE 3 — Model Architecture
--------------------------------------------------------

Which model to choose:

  OPTION A — Fine-tune SecureBERT or DistilBERT (our plan)
    Model: "ehsanaghaei/SecureBERT" on HuggingFace
    Input: raw URL string (tokenized automatically)
    Output: 3-class softmax [safe, suspicious, dangerous]
    Size: ~110MB before quantization, ~30MB after ONNX INT8 quantization
    Expected F1: 97–99%
    Best for: API deployment, publishable model

  OPTION B — Character-level CNN (alternative, smaller)
    Architecture: Embedding → Conv1D × 3 → GlobalMaxPool → Dense → Softmax
    Size: ~1–2MB (fits in browser)
    Expected F1: 94–97%
    Best for: in-browser use

  OPTION C — Fine-tune large → distil into small (advanced)
    Train BERT first → knowledge distillation → tiny student model
    Best of both worlds but more complex
    Best for: long-term production use

How fine-tuning SecureBERT works:
  URL string → tokenizer → BERT 12 layers of attention → [CLS] embedding (768 dims)
             → classification head (2 linear layers added on top) → [safe, susp, danger]

  "Fine-tuning" = BERT already knows language patterns from billions of words.
  We keep all that knowledge and add a small classification head, then train the
  whole thing on our URL data. Much faster than training from scratch.

Key hyperparameters:
  LEARNING_RATE  = 2e-5    # How big a step per weight update. Range for BERT: 1e-5 to 5e-5
  BATCH_SIZE     = 32      # URLs processed before each weight update. Common: 16, 32, 64
  EPOCHS         = 3–5     # Full passes through training data. BERT rarely needs more than 5
  MAX_LENGTH     = 128     # Max URL tokens (truncate longer). Covers 95%+ of real URLs
  WARMUP_STEPS   = 10%     # Gradually raise LR for first 10% of training steps

--------------------------------------------------------
PHASE 4 — Training
--------------------------------------------------------

The training loop (what happens each step):
  1. Take a batch of URLs + their labels
  2. Run through model → get predictions
  3. Calculate loss: how wrong were predictions? (CrossEntropyLoss)
  4. Backpropagate: calculate how to adjust each weight to reduce loss
  5. Optimizer step: actually adjust the weights
  6. Repeat

Essential training practices:

  Learning rate scheduler (don't use constant LR):
    Linear warmup → linear decay
    Standard for BERT fine-tuning
    Use: from transformers import get_linear_schedule_with_warmup

  Gradient clipping (prevents exploding gradients):
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

  Mixed precision training (almost free 2x speedup):
    from torch.cuda.amp import autocast, GradScaler
    Runs float16 where possible, float32 where needed

  Checkpoint saving (save model every epoch):
    Save: model weights, optimizer state, epoch number, best val_f1
    Always save the BEST checkpoint = highest val_F1, NOT necessarily last epoch
    torch.save(checkpoint, f'checkpoint_epoch_{epoch}.pt')

Experiment tracking with WandB (non-negotiable):
  import wandb
  wandb.init(project="linkguard-url-classifier")
  wandb.log({
      "train/loss": loss,
      "train/accuracy": acc,
      "val/loss": val_loss,
      "val/f1": val_f1,
      "epoch": epoch,
      "learning_rate": scheduler.get_last_lr()[0]
  })
  WandB gives live graphs, run comparison, reproducibility, and public sharing.
  The HuggingFace model card will link to these training curves.

What healthy training looks like:
  Epoch 1: train_loss=0.45, val_loss=0.38, val_f1=0.91  ← improving
  Epoch 2: train_loss=0.28, val_loss=0.24, val_f1=0.96  ← improving
  Epoch 3: train_loss=0.18, val_loss=0.22, val_f1=0.97  ← slightly improving
  Epoch 4: train_loss=0.12, val_loss=0.25, val_f1=0.96  ← STOP, overfitting starts

Warning signs and fixes:
  Loss is NaN              → LR too high. Restart with 10x lower LR
  Loss not decreasing      → LR too low or data pipeline bug. Check data loading
  Val loss rises, train ok → Overfitting. Reduce epochs, add dropout, get more data
  GPU memory error         → Batch size too large. Halve it
  Training very slow       → Not using GPU / mixed precision. Check CUDA, enable fp16

PHASE 5 — Evaluation
--------------------------------------------------------

Metrics to use (accuracy alone is misleading with imbalanced classes):

  Precision = of all URLs I flagged as dangerous, how many actually were?
              High precision = few false alarms

  Recall    = of all actually dangerous URLs, how many did I catch?
              High recall = few missed threats
              *** For security tools, RECALL matters more than precision ***
              Missing a dangerous URL is worse than a false alarm.

  F1 Score  = harmonic mean of precision and recall (the main metric to optimize)

  AUC-ROC   = model's ability to distinguish classes regardless of threshold
              1.0 = perfect, 0.5 = random guessing

Confusion matrix (most important visualization):
                Predicted
                Safe   Susp   Danger
  Actual Safe  [9850    80      70]   ← 150 false alarms (acceptable)
        Susp   [ 120   780     100]
        Danger [  20    15    9965]   ← 35 missed threats (minimize this)

  Low numbers off the diagonal = good model.

Error analysis (do this manually — critical):
  Take the 50 worst wrong predictions. Look at them.
  Example pattern to find:
    Predicted SAFE, actually DANGEROUS:
      - "google-secure.accountverify.xyz/login"  ← mimics Google
      - "bit.ly/3xK9mP"  ← URL shortener hides destination
    Predicted DANGEROUS, actually SAFE:
      - "testphishing.com"  ← security researcher test site
  This reveals what to improve next.

Benchmark against existing tools:
  Test on URLs that VirusTotal flags. Goal: catch ≥85% of what VT catches,
  plus catch novel/zero-day URLs that VT misses.

--------------------------------------------------------
MOST COMMON BEGINNER MISTAKES (avoid these)
--------------------------------------------------------

1. Evaluating on test set during development — use val set until you are done
2. Not logging experiments to WandB — you will lose track of what worked
3. Training too many epochs — BERT fine-tuning rarely needs more than 5
4. Ignoring class imbalance — model silently fails on minority classes
5. Not doing error analysis — metrics lie, looking at failures tells the truth
6. Skipping the model card — nobody will use a model with no documentation
7. Not versioning data — always know exactly which data produced which model
